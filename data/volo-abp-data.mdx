`Volo.Abp.Data` defines abstractions the rest of the data stack depends on. It has **no** dependency on EF Core, MongoDB, Dapper or any driver. `AbpDataModule` itself depends on `AbpObjectExtendingModule`, `AbpUnitOfWorkModule` and `AbpEventBusAbstractionsModule`; every persistence-provider package (`Volo.Abp.EntityFrameworkCore`, `Volo.Abp.MongoDB`, `Volo.Abp.Dapper`, `Volo.Abp.MemoryDb`) sits on top of it through `AbpDddDomainModule`. It feeds the multi-tenancy and migration subsystems.

Source: `framework/src/Volo.Abp.Data/Volo/Abp/Data/`.

## Module — `AbpDataModule`

File: `AbpDataModule.cs`.

```csharp
[DependsOn(
    typeof(AbpObjectExtendingModule),
    typeof(AbpUnitOfWorkModule),
    typeof(AbpEventBusAbstractionsModule)
)]
public class AbpDataModule : AbpModule { ... }
```

`AbpDataModule.PreConfigureServices` calls private `AutoAddDataSeedContributors(context.Services)`, which registers a `services.OnRegistered` callback that inspects every subsequent service registration and, for those whose `ImplementationType` is assignable to `IDataSeedContributor`, appends the type to `AbpDataSeedOptions.Contributors` via `services.Configure<AbpDataSeedOptions>(...)`. `ConfigureServices` binds `AbpDbConnectionOptions` from `IConfiguration` (so `appsettings.json`'s `ConnectionStrings` section is auto-wired) and registers the open generic singleton `IDataFilter<>` → `DataFilter<>`. `PostConfigureServices` calls `Configure<AbpDbConnectionOptions>(o => o.Databases.RefreshIndexes())` so `AbpDatabaseInfoDictionary.GetMappedDatabaseOrNull` is O(1) after module setup.

## Data filters

| File | Type | Notes |
| --- | --- | --- |
| `IDataFilter.cs` | `IDataFilter` (non-generic) and `IDataFilter<TFilter>` | `IDataFilter<TFilter>` exposes `IDisposable Enable()`, `IDisposable Disable()`, `bool IsEnabled`. `IDataFilter` is the type-routed façade with `Enable<TFilter>()`, `Disable<TFilter>()`, `IsEnabled<TFilter>()`. |
| `DataFilter.cs` | `DataFilter : IDataFilter, ISingletonDependency` + `DataFilter<TFilter> : IDataFilter<TFilter>` | `DataFilter` resolves `IDataFilter<TFilter>` from DI and caches it in a `ConcurrentDictionary<Type, object>`. `DataFilter<TFilter>` holds its state in an `AsyncLocal<DataFilterState>`, lazy-initialized from `AbpDataFilterOptions.DefaultStates[typeof(TFilter)]?.Clone()` (defaulting to `new DataFilterState(true)`), so the toggle is per async-flow and does not leak across requests. |
| `DataFilterState.cs` | `DataFilterState` | Mutable `bool IsEnabled` with a `Clone()` that returns `new DataFilterState(IsEnabled)`. |
| `AbpDataFilterOptions.cs` | `AbpDataFilterOptions` | `DefaultStates : Dictionary<Type, DataFilterState>` — set per-filter default during module configuration. Filters not present in the dictionary default to enabled. |
| `DataFilterExtensions.cs` | extensions | Variadic `Disable<T1,T2>()` … `Disable<T1,T2,T3,T4,T5,T6,T7,T8>()` and the matching `Enable<T1,T2>()` … `Enable<T1,…,T8>()` that each return a private `CompositeDisposable` re-enabling/disabling every filter on dispose. |

The default registration is in `AbpDataModule.ConfigureServices`:

```csharp
context.Services.AddSingleton(typeof(IDataFilter<>), typeof(DataFilter<>));
```

EF Core wires the filters into global query filters via `AbpDbContext.ShouldFilterEntity<TEntity>` and `CreateFilterExpression<TEntity>` (see `/data/data-filtering`).

## Data seeding

| File | Type | Notes |
| --- | --- | --- |
| `IDataSeeder.cs` | `IDataSeeder` | `Task SeedAsync(DataSeedContext)`. |
| `DataSeeder.cs` | `DataSeeder : IDataSeeder, ITransientDependency` | The default implementation. `[UnitOfWork]` attribute on `SeedAsync`. Creates an `IServiceScope`, loops `AbpDataSeedOptions.Contributors`, and optionally opens a separate UoW per contributor (driven by `DataSeederExtensions.SeedInSeparateUow` properties on the context). |
| `IDataSeedContributor.cs` | `IDataSeedContributor` | The hook each module implements. |
| `DataSeedContext.cs` | `DataSeedContext` | `TenantId?`, `Properties : Dictionary<string,object?>`, indexer, `WithProperty(name, value)` fluent builder. |
| `DataSeederExtensions.cs` | extensions | `SeedAsync(this IDataSeeder, Guid? tenantId)` + `SeedInSeparateUowAsync(..., AbpUnitOfWorkOptions, bool requiresNew)`. Defines the constants `SeedInSeparateUow`, `SeedInSeparateUowOptions`, `SeedInSeparateUowRequiresNew`. |
| `AbpDataSeedOptions.cs` | `AbpDataSeedOptions` | Single property `Contributors : DataSeedContributorList`. |
| `DataSeedContributorList.cs` | `DataSeedContributorList : TypeList<IDataSeedContributor>` | Ordered list of contributor types. |

`AbpDataModule.AutoAddDataSeedContributors` uses `services.OnRegistered` to enroll every contributor automatically, so application code only needs to register a class implementing `IDataSeedContributor` — no manual hookup. The DbMigrator template then calls `IDataSeeder.SeedAsync(tenantId)`.
