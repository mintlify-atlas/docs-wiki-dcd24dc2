`Volo.Abp.Uow` is the orchestrator for transactional database access and deferred event publishing. Every `[UnitOfWork]`-attributed method, every application service, every `IDataSeedContributor` and every controller action that calls a repository runs inside an `IUnitOfWork`.

Source: `framework/src/Volo.Abp.Uow/Volo/Abp/Uow/`.

## Module

`AbpUnitOfWorkModule` (file `AbpUnitOfWorkModule.cs`) hooks `UnitOfWorkInterceptorRegistrar.RegisterIfNeeded` into `IServiceCollection.OnRegistered`. Any registered type whose declaring class implements `IUnitOfWorkEnabled` or carries `[UnitOfWork]` gets wrapped with `UnitOfWorkInterceptor`.

## Lifecycle

```mermaid
sequenceDiagram
    autonumber
    participant Caller as Caller (controller / app service)
    participant Int as UnitOfWorkInterceptor
    participant Mgr as UnitOfWorkManager
    participant Amb as AmbientUnitOfWork (AsyncLocal)
    participant Uow as UnitOfWork
    participant DB as DatabaseApi(s)
    participant Tx as TransactionApi(s)
    participant Pub as IUnitOfWorkEventPublisher

    Caller->>Int: invoke virtual method
    Int->>Mgr: TryBeginReserved(_AbpActionUnitOfWork) or Begin(options)
    Mgr->>Uow: new UnitOfWork() via scope.ServiceProvider
    Mgr->>Amb: SetUnitOfWork(uow)
    Mgr-->>Int: IUnitOfWork
    Int->>Caller: ProceedAsync()
    Caller->>Uow: implicit via IDbContextProvider / repository
    Uow->>DB: AddDatabaseApi("EntityFrameworkCore:<TDbContext>", api)
    alt IsTransactional
        Uow->>Tx: AddTransactionApi(...)
    end
    Caller-->>Int: returns
    Int->>Uow: CompleteAsync()
    Uow->>Uow: SaveChangesAsync() → DB.SaveChangesAsync()
    Uow->>Pub: PublishLocalEventsAsync(LocalEvents) ordered by EventOrder
    Uow->>Pub: PublishDistributedEventsAsync(DistributedEvents)
    Uow->>Tx: CommitAsync() on every transaction api
    Uow-->>Int: completed
    Int->>Uow: Dispose() (Disposed event fires; Amb pops)
```

### Outer / inner / reserved

- **Outer UoW exists & `requiresNew == false`** → `Begin` returns a `ChildUnitOfWork`. The child's `CompleteAsync` is a no-op; only the outer commits.
- **Outer UoW exists & `requiresNew == true`** → a fresh root UoW is created, `SetOuter(outerUow)` is called so it can still chain `Failed` events. The new UoW becomes the ambient one until disposed.
- **Reservation** — `Reserve("_AbpActionUnitOfWork")` creates a UoW with `IsReserved = true` early in the pipeline (e.g. `AbpUnitOfWorkMiddleware` for ASP.NET Core). `[UnitOfWork]`-wrapped methods then call `TryBeginReserved(name, options)` which initialises the reserved UoW with the method's options, so middleware-level state (current tenant, current user) is captured before the action runs.

### Event ordering

`UnitOfWork.AddOrReplaceLocalEvent` / `AddOrReplaceDistributedEvent` accept an optional `Predicate<UnitOfWorkEventRecord>? replacementSelector`. If supplied and a previous record matches, the new event replaces it — this is how "entity updated" events deduplicate across multiple modifications. Otherwise the event is appended with `EventOrderGenerator.GetNext()`. On commit, events are sorted by `EventOrder` ascending, then dispatched via `IUnitOfWorkEventPublisher`.

### Transaction decision

`UnitOfWorkInterceptor.CreateOptions` resolves `IsTransactional` like this when the attribute does not set it:

```csharp
options.IsTransactional = defaultOptions.CalculateIsTransactional(
    autoValue: serviceProvider.GetRequiredService<IUnitOfWorkTransactionBehaviourProvider>().IsTransactional
               ?? !invocation.Method.Name.StartsWith("Get", StringComparison.InvariantCultureIgnoreCase)
);
```

So by default a method whose name starts with `Get` is read-only (no transaction), everything else opens one — unless `AbpUnitOfWorkDefaultOptions.TransactionBehavior` is `Enabled`/`Disabled` to force it.

## Failure handling

`UnitOfWork.SaveChangesAsync` / `CompleteAsync` catch exceptions, stash them in `_exception`, raise `Failed`, then rethrow. `Dispose` rolls back any active transaction (`ISupportsRollback`), fires `Disposed`, and pops itself off `AmbientUnitOfWork`.

`AbpDbConcurrencyException` (in `Volo.Abp.Data`) is the canonical concurrency error — EF Core wraps `DbUpdateConcurrencyException` into it via `AbpDbContext.SaveChangesOnDbContextAsync`.
