Each EF Core dialect lives in its own NuGet package under `framework/src/`. Every package follows the same skeleton:

- `Volo/Abp/EntityFrameworkCore/<Dialect>/AbpEntityFrameworkCore<Dialect>Module.cs` — `AbpModule` that depends on `AbpEntityFrameworkCoreModule`, sets the sequential GUID type appropriate for the database, and flips `AbpEfCoreGlobalFilterOptions.UseDbFunction = true`.
- `Volo/Abp/EntityFrameworkCore/AbpDbContextOptions<Dialect>Extensions.cs` — `Use<Dialect>()` extension on `AbpDbContextOptions`.
- `Volo/Abp/EntityFrameworkCore/AbpDbContextConfigurationContext<Dialect>Extensions.cs` — `Use<Dialect>()` extension on `AbpDbContextConfigurationContext` (called inside `options.Configure(...)`).
- `Volo/Abp/EntityFrameworkCore/ConnectionStrings/<Dialect>ConnectionStringChecker.cs` — `IConnectionStringChecker` implementation that opens a 1-second-timeout connection.
- `Microsoft/EntityFrameworkCore/Abp<Dialect>ModelBuilderExtensions.cs` (where applicable) — provider-specific model-builder helpers.

## Matrix

| NuGet package | Module class | `AbpDbContextOptions` extension | Underlying provider package | Default sequential GUID |
| --- | --- | --- | --- | --- |
| `Volo.Abp.EntityFrameworkCore.SqlServer` | `AbpEntityFrameworkCoreSqlServerModule` | `UseSqlServer(Action<SqlServerDbContextOptionsBuilder>?)` | `Microsoft.EntityFrameworkCore.SqlServer` | `SequentialAtEnd` |
| `Volo.Abp.EntityFrameworkCore.PostgreSql` | `AbpEntityFrameworkCorePostgreSqlModule` | `UseNpgsql(Action<NpgsqlDbContextOptionsBuilder>?)` (also legacy `UsePostgreSql` marked `[Obsolete]`) | `Npgsql.EntityFrameworkCore.PostgreSQL` | `SequentialAsString` |
| `Volo.Abp.EntityFrameworkCore.MySQL` | `AbpEntityFrameworkCoreMySQLModule` | `UseMySQL(Action<MySQLDbContextOptionsBuilder>?)` | `MySql.EntityFrameworkCore` (Oracle's) | `SequentialAsString` |
| `Volo.Abp.EntityFrameworkCore.MySQL.Pomelo` | `AbpEntityFrameworkCoreMySQLPomeloModule` | `UseMySQL(Action<MySqlDbContextOptionsBuilder>?)` | `Pomelo.EntityFrameworkCore.MySql` | `SequentialAsString` |
| `Volo.Abp.EntityFrameworkCore.Sqlite` | `AbpEntityFrameworkCoreSqliteModule` | `UseSqlite(Action<SqliteDbContextOptionsBuilder>?)` | `Microsoft.EntityFrameworkCore.Sqlite` | — (Sqlite module does **not** set `DefaultSequentialGuidType`; framework default applies) |
| `Volo.Abp.EntityFrameworkCore.Oracle` | `AbpEntityFrameworkCoreOracleModule` | `UseOracle(Action<OracleDbContextOptionsBuilder>?)` | `Oracle.EntityFrameworkCore` | `SequentialAsBinary` |
| `Volo.Abp.EntityFrameworkCore.Oracle.Devart` | `AbpEntityFrameworkCoreOracleDevartModule` | `UseOracle(Action<OracleDbContextOptionsBuilder>? = null, bool useExistingConnectionIfAvailable = false)` (builder from `Devart.Data.Oracle.Entity`) | `Devart.Data.Oracle.EFCore` | `SequentialAsBinary` |

Every dialect module — except Sqlite, see below — looks essentially like the SQL Server one:

```csharp
[DependsOn(typeof(AbpEntityFrameworkCoreModule))]
public class AbpEntityFrameworkCoreSqlServerModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        Configure<AbpSequentialGuidGeneratorOptions>(options =>
        {
            if (options.DefaultSequentialGuidType == null)
            {
                options.DefaultSequentialGuidType = SequentialGuidType.SequentialAtEnd;
            }
        });
        Configure<AbpEfCoreGlobalFilterOptions>(options =>
        {
            options.UseDbFunction = true;
        });
    }
}
```

The differences are: SQL Server uses `SequentialAtEnd` (matches `NEWSEQUENTIALID()`'s byte ordering); PostgreSQL and both MySQL packages use `SequentialAsString` (text-ordered GUIDs sort well in `uuid` and `char(36)`); Oracle and Oracle.Devart use `SequentialAsBinary` (raw `RAW(16)` binary order). Sqlite does not configure `AbpSequentialGuidGeneratorOptions` at all — first match wins, so whichever dialect module is loaded sets the type. All seven dialect modules set `AbpEfCoreGlobalFilterOptions.UseDbFunction = true`.

## `UseSqlServer` (`Volo.Abp.EntityFrameworkCore.SqlServer`)

`AbpDbContextOptionsSqlServerExtensions.UseSqlServer(options, action)` calls `options.Configure(ctx => ctx.UseSqlServer(action))`. The inner `AbpDbContextConfigurationContextSqlServerExtensions.UseSqlServer` checks `context.ExistingConnection` and chooses between `context.DbContextOptions.UseSqlServer(existingConnection, …)` and `…UseSqlServer(connectionString, …)`. In **both** branches it sets `optionsBuilder.UseQuerySplittingBehavior(QuerySplittingBehavior.SplitQuery)` before invoking the caller-supplied `sqlServerOptionsAction` — split-query is the ABP default to avoid cartesian explosions with `Include` chains.

`SqlServerConnectionStringChecker` (in `ConnectionStrings/`) builds a `SqlConnectionStringBuilder` with `ConnectTimeout = 1`, swaps `InitialCatalog` to `master` to ensure the open succeeds even when the target catalog is missing, then `ChangeDatabaseAsync(oldDatabaseName)` to verify the catalog exists — setting `Connected = true` after open and `DatabaseExists = true` after the change. Registered with `[Dependency(ReplaceServices = true), ITransientDependency]`, so referencing the SqlServer package automatically swaps `DefaultConnectionStringChecker`.

## `UseNpgsql` (`Volo.Abp.EntityFrameworkCore.PostgreSql`)

```csharp
public static void UseNpgsql(
    [NotNull] this AbpDbContextOptions options,
    Action<NpgsqlDbContextOptionsBuilder>? postgreSqlOptionsAction = null)
{
    options.Configure(context => context.UseNpgsql(postgreSqlOptionsAction));
}
```

Both `UsePostgreSql(options, action)` and `UsePostgreSql<TDbContext>(options, action)` are kept as `[Obsolete("Use 'UseNpgsql(...)' method instead. This will be removed in future versions.")]` shims that forward to `context.UseNpgsql(...)`. `NpgsqlConnectionStringChecker` swaps the database to `postgres` (the always-present default catalog on a PostgreSQL server) with `Timeout = 1`, opens the connection, then calls `ChangeDatabaseAsync(oldDatabaseName)` to confirm the target database exists. `Microsoft/EntityFrameworkCore/AbpPostgreSqlModelBuilderExtensions.cs` exposes provider-specific model-builder helpers.

## `UseMySQL` (Oracle's driver — `Volo.Abp.EntityFrameworkCore.MySQL`)

Uses `MySql.EntityFrameworkCore.Infrastructure.MySQLDbContextOptionsBuilder`. Checker: `MySQLConnectionStringChecker`.

## `UseMySQL` (Pomelo — `Volo.Abp.EntityFrameworkCore.MySQL.Pomelo`)

Same method name `UseMySQL`, but the builder type is `Microsoft.EntityFrameworkCore.Infrastructure.MySqlDbContextOptionsBuilder` (Pomelo). Pomelo's `UseMySql` requires a server version — the configuration context extension handles auto-detection. Checker: `PomeloMySQLConnectionStringChecker`.

> Don't reference both `Volo.Abp.EntityFrameworkCore.MySQL` and `.MySQL.Pomelo` in the same project — both expose `UseMySQL` with conflicting signatures.

## `UseSqlite` (`Volo.Abp.EntityFrameworkCore.Sqlite`)

```csharp
public static void UseSqlite(
    [NotNull] this AbpDbContextOptions options,
    Action<SqliteDbContextOptionsBuilder>? sqliteOptionsAction = null)
{
    options.Configure(context => context.UseSqlite(sqliteOptionsAction));
}
```

Sqlite is the odd module out — it has both `PreConfigureServices` and `ConfigureServices`:

```csharp
public override void PreConfigureServices(ServiceConfigurationContext context)
{
    PreConfigure<AbpSqliteOptions>(options =>
    {
        options.BusyTimeout = 5000;
    });
}

public override void ConfigureServices(ServiceConfigurationContext context)
{
    Configure<AbpEfCoreGlobalFilterOptions>(options => { options.UseDbFunction = true; });

    var sqliteOptions = context.Services.ExecutePreConfiguredActions<AbpSqliteOptions>();
    if (sqliteOptions.BusyTimeout.HasValue)
    {
        Configure<AbpDbContextOptions>(options =>
        {
            options.ConfigureDefaultOnConfiguring((dbContext, dbContextOptionsBuilder) =>
            {
                if (dbContextOptionsBuilder.Options.Extensions.Any(e => e is SqliteOptionsExtension))
                {
                    dbContextOptionsBuilder.AddInterceptors(
                        new SqliteBusyTimeoutSaveChangesInterceptor(sqliteOptions.BusyTimeout.Value));
                }
            }, overrideExisting: false);
        });
    }
}
```

Extras specific to SQLite:

- `AbpSqliteOptions.BusyTimeout` (milliseconds, default `5000`) — overridable in another module's `PreConfigureServices` via `PreConfigure<AbpSqliteOptions>`.
- `Interceptors/SqliteBusyTimeoutSaveChangesInterceptor.cs` — `SaveChangesInterceptor` that issues `PRAGMA busy_timeout` on the underlying SQLite connection. Attached only to `DbContext`s that already carry a `SqliteOptionsExtension`.
- `Sqlite/AbpUnitTestSqliteConnection.cs` — helper for unit tests that keeps a single `:memory:` SQLite connection open across the lifetime of a test fixture.
- Checker: `SqliteConnectionStringChecker` — opens the file (or `:memory:`) via `SqliteConnection.OpenAsync`. SQLite creates the file if missing, so a successful open implies `Connected = true, DatabaseExists = true`.

## `UseOracle` (`Volo.Abp.EntityFrameworkCore.Oracle`)

Uses `Oracle.EntityFrameworkCore.Infrastructure.OracleDbContextOptionsBuilder`. Checker: `OracleConnectionStringChecker`.

## `UseOracle` (Devart — `Volo.Abp.EntityFrameworkCore.Oracle.Devart`)

```csharp
using Devart.Data.Oracle.Entity;

public static void UseOracle(
    [NotNull] this AbpDbContextOptions options,
    Action<OracleDbContextOptionsBuilder>? oracleOptionsAction = null,
    bool useExistingConnectionIfAvailable = false)
{
    options.Configure(context =>
    {
        context.UseOracle(oracleOptionsAction, useExistingConnectionIfAvailable);
    });
}
```

Adds the `useExistingConnectionIfAvailable` flag — Devart can reuse an already-open `DbConnection` instead of opening a fresh one, useful when `AbpDbContextConfigurationContext.ExistingConnection` is set (e.g. in tests that share a connection). Checker: `OracleDevartConnectionStringChecker`. The Devart package also ships `Microsoft/EntityFrameworkCore/AbpOracleModelBuilderExtensions.cs` for Oracle-specific column type mapping; the non-Devart `Volo.Abp.EntityFrameworkCore.Oracle` package does not ship a model-builder extensions file.

## Choosing a sequential GUID

`AbpSequentialGuidGeneratorOptions.DefaultSequentialGuidType` is set on first match, never overwritten (`options.DefaultSequentialGuidType ??= ...`). If you mix providers (rare), the first registered dialect wins. The values come from `Volo.Abp.Guids.SequentialGuidType`:

| Value | Picks contiguous bytes at | Best for |
| --- | --- | --- |
| `SequentialAsString` | end (string-sortable) | Postgres `uuid`, MySQL `char(36)`. |
| `SequentialAsBinary` | beginning (binary-sortable) | Oracle `RAW(16)`. |
| `SequentialAtEnd` | end (SQL Server's specific `uniqueidentifier` byte order) | SQL Server. |

## Composition example

```csharp
[DependsOn(typeof(AbpEntityFrameworkCoreSqlServerModule))]
public class MyAppEntityFrameworkCoreModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        context.Services.AddAbpDbContext<MyAppDbContext>(options =>
        {
            options.AddDefaultRepositories(includeAllEntities: true);
        });

        Configure<AbpDbContextOptions>(options =>
        {
            options.UseSqlServer();  // ← provider extension
        });
    }
}
```

The connection string is resolved via `IConnectionStringResolver` — the lookup key is `ConnectionStringNameAttribute.GetConnStringName(typeof(MyAppDbContext))`, which defaults to the full type name when no attribute is present.
