description: "Pluggable IObjectValidator pipeline, DataAnnotations contributor, the [UnitOfWork]-style ValidationInterceptor, and AbpValidationException."
---

`Volo.Abp.Validation` validates *objects* (DTOs) and *method invocations*. It is a contributor pipeline: `IObjectValidator` enumerates registered `IObjectValidationContributor`s, collects `ValidationResult`s, and throws `AbpValidationException` if any are produced. `ValidationInterceptor` runs the same pipeline for every public method on every class that implements `IValidationEnabled` — application services are validated automatically.

Source: `framework/src/Volo.Abp.Validation/Volo/Abp/Validation/` and `framework/src/Volo.Abp.Validation.Abstractions/Volo/Abp/Validation/`.

## Modules

`AbpValidationAbstractionsModule` (in `Volo.Abp.Validation.Abstractions`) ships only `AbpValidationException`, `IHasValidationErrors`, and `ValidationHelper.EmailRegEx` / `IsValidEmailAddress(string)`. Application contracts depend on this so they can throw validation errors without pulling the interceptor.

`AbpValidationModule` (in `Volo.Abp.Validation`) depends on `AbpValidationAbstractionsModule` and `AbpLocalizationModule`. In `PreConfigureServices` it:

1. Hooks `ValidationInterceptorRegistrar.RegisterIfNeeded` into `IServiceCollection.OnRegistered` so types implementing `IValidationEnabled` are wrapped with `ValidationInterceptor`.
2. Scans every registered type for `IObjectValidationContributor` and adds it to `AbpValidationOptions.ObjectValidationContributors`.

In `ConfigureServices` it embeds `AbpValidationResource` localization files into the virtual file system and adds the English-default resource to `AbpLocalizationOptions.Resources`.

## Validation flow

1. A request hits `MyAppService.CreateAsync(MyDto input)`.
2. `MyAppService : ApplicationService` implements `IValidationEnabled`, so the registrar has wired `ValidationInterceptor`.
3. `ValidationInterceptor.InterceptAsync` builds a `MethodInvocationValidationContext` and calls `IMethodInvocationValidator.ValidateAsync`.
4. `MethodInvocationValidator` short-circuits when `Parameters.Length == 0`, the method isn't public, or it sees `[DisableValidation]`. Otherwise it iterates parameters and calls `IObjectValidator.GetErrorsAsync(value, name, allowNulls)`.
5. `ObjectValidator` creates an `ObjectValidationContext` and invokes every `IObjectValidationContributor`. The built-in `DataAnnotationObjectValidationContributor` runs `ValidationAttribute`s and any `IValidatableObject.Validate` on the DTO recursively.
6. If `FluentObjectValidationContributor` is registered (see `/crosscut/fluent-validation`), it asks DI for `IValidator<MyDto>` and adds FluentValidation errors.
7. Errors are aggregated into the `MethodInvocationValidationContext.Errors`. If any are present, `AbpValidationException("Method arguments are not valid! ...", errors)` is thrown before the method runs.
8. The exception is caught by `AbpExceptionFilter` / `AbpExceptionHandlingMiddleware`, formatted into a `RemoteServiceErrorResponse` with the localized messages, and returned as HTTP 400.

## Throwing manually

```csharp
throw new AbpValidationException(
    "Custom validation failed!",
    new List<ValidationResult>
    {
        new ValidationResult("Username already exists.", new[] { nameof(input.UserName) })
    });
```

Because `AbpValidationException : IExceptionWithSelfLogging`, the framework logs each member error at `LogLevel.Warning` when the exception bubbles out.

## Localizing messages

Pass `ErrorMessageResourceType` / `ErrorMessageResourceName` to your DataAnnotations attributes, or implement a custom `IAttributeValidationResultProvider` that resolves the message through `IStringLocalizer`. The default exception handler picks them up.
