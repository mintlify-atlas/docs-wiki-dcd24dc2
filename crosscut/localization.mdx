---
title: "Localization (Volo.Abp.Localization)"
description: "Dictionary-backed IStringLocalizer over LocalizationResource, JSON virtual files, inherited resources, base-culture fallback, and the external/dynamic resource store."
---

`Volo.Abp.Localization` replaces ASP.NET Core's `ResourceManagerStringLocalizerFactory` with a dictionary-based `AbpStringLocalizerFactory` that resolves strings from `LocalizationResource`s. Each resource is a *class* (a marker type) plus one or more `ILocalizationResourceContributor`s that load JSON files out of the virtual file system. Resources can inherit from one another, fall back along culture chains (`tr-TR → tr → en`), and be extended at runtime via `IExternalLocalizationStore`.

Source: `framework/src/Volo.Abp.Localization/Volo/Abp/Localization/` and `framework/src/Volo.Abp.Localization.Abstractions/Volo/Abp/Localization/`.

## Modules

`AbpLocalizationAbstractionsModule` (in `Volo.Abp.Localization.Abstractions`) ships only the `ILocalizableString` family, `LocalizableString`, `FixedLocalizableString`, `IAsyncLocalizableString`, `LocalizationResourceNameAttribute`, `IAbpStringLocalizerFactory` (the async extension over `IStringLocalizerFactory`), and `LocalizableStringExtensions`. Domain-shared and contracts projects depend on this so they can carry localizable strings without needing the resource pipeline.

`AbpLocalizationModule` depends on `AbpVirtualFileSystemModule`, `AbpSettingsModule`, `AbpLocalizationAbstractionsModule`, `AbpThreadingModule`. In `ConfigureServices` it:

1. Replaces `IStringLocalizerFactory` with `AbpStringLocalizerFactory` (via `AbpStringLocalizerFactory.Replace(services)`).
2. Embeds its own assembly into `AbpVirtualFileSystemOptions.FileSets`.
3. Configures `AbpLocalizationOptions.Resources` to include the framework's own `DefaultResource` and `AbpLocalizationResource` with `AddVirtualJson("/Localization/Resources/AbpLocalization")`.

## `AbpLocalizationOptions`

| Property | Type | Role |
| --- | --- | --- |
| `Resources` | `LocalizationResourceDictionary` | The catalog of `LocalizationResource`s keyed by name. Mutated via `Add<TResource>(defaultCulture)` and chained with `.AddVirtualJson(path)` / `.AddBaseTypes(...)`. |
| `DefaultResourceType` | `Type?` | When `LocalizableString` has no specific resource attached, this is used. Falls back to `DefaultResource` when null. |
| `GlobalContributors` | `ITypeList<ILocalizationResourceContributor>` | Contributors attached to *every* resource (e.g. dynamic translations). Added when each resource's `AbpDictionaryBasedStringLocalizer` is constructed. |
| `Languages` | `List<LanguageInfo>` | The languages the host advertises (`CultureName`, `UiCultureName`, `DisplayName`, `TwoLetterISOLanguageName`). Drives `DefaultLanguageProvider`. |
| `LanguagesMap`, `LanguageFilesMap` | `Dictionary<string, List<NameValue>>` | Per-module culture → physical-file name mapping. Used by the localization-key synchronizer / dictionary builders. |
| `TryToGetFromBaseCulture` | `bool` (default `true`) | If `tr-TR` is missing a key, try `tr`. |
| `TryToGetFromDefaultCulture` | `bool` (default `true`) | If `tr` is still missing, try the resource's `DefaultCultureName`. |

## Defining a resource

```csharp
[LocalizationResourceName("MyApp")]
public class MyAppResource
{
}
```

Register it once in your module's `ConfigureServices`:

```csharp
Configure<AbpLocalizationOptions>(options =>
{
    options.Resources
        .Add<MyAppResource>("en")
        .AddVirtualJson("/Localization/MyApp")
        .AddBaseTypes(typeof(AbpUiResource));   // inherit translations
});

Configure<AbpVirtualFileSystemOptions>(options =>
{
    options.FileSets.AddEmbedded<MyAppModule>();
});
```

Place JSON files under `Localization/MyApp/en.json`, `tr.json`, … and embed them. Each file looks like:

```json
{
  "culture": "en",
  "texts": {
    "Welcome": "Welcome to MyApp",
    "Greeting": "Hello, {0}!"
  }
}
```

`JsonLocalizationFile` has only two fields — `Culture` (string) and `Texts` (dictionary). `JsonLocalizationDictionaryBuilder.BuildFromJsonString` flattens nested objects/arrays into `Parent__Child__0` keys using the `__` separator. Duplicates throw `AbpException`.

## Using strings in code

The factory is just `IStringLocalizerFactory` — inject `IStringLocalizer<MyAppResource>` directly:

```csharp
public class HelloService : ITransientDependency
{
    private readonly IStringLocalizer<MyAppResource> _l;
    public HelloService(IStringLocalizer<MyAppResource> l) => _l = l;

    public string Greet(string name) => _l["Greeting", name];
}
```

For abstractions and domain-shared code that can't depend on `IStringLocalizer` directly, use `LocalizableString`:

```csharp
var s = LocalizableString.Create<MyAppResource>("Welcome");
// later, in a service that has IStringLocalizerFactory:
var localized = s.Localize(stringLocalizerFactory);     // sync
var localized2 = await s.LocalizeAsync(stringLocalizerFactory); // async — needed for dynamic/external stores
```

`LocalizableString` stores `ResourceType?` and/or `ResourceName?` plus the `Name` key. If both lookups miss, it falls back to `IStringLocalizerFactory.CreateDefaultOrNull()` (i.e. `AbpLocalizationOptions.DefaultResourceType`). `FixedLocalizableString(string value)` is the constant-string escape hatch — it returns its own value unchanged.

## Resource resolution order

For `AbpDictionaryBasedStringLocalizer[name]` with culture `c1` and the resource's `DefaultCultureName == "en"`:

1. `Resource.Contributors.GetOrNull(c1, name)` — original culture.
2. If `c1` contains `-` and `TryToGetFromBaseCulture`, `Resource.Contributors.GetOrNull(baseCulture(c1), name)`.
3. If `TryToGetFromDefaultCulture` and `Resource.DefaultCultureName` is set, `Resource.Contributors.GetOrNull(DefaultCultureName, name)`.
4. For each inherited resource in `BaseLocalizers`, `(baseLocalizer)[name]` under `CultureHelper.Use(...)`.
5. Otherwise `new LocalizedString(name, name, resourceNotFound: true)` — the raw key is returned.

Inside `Resource.Contributors`, lookups iterate **in reverse** insertion order, so later-added contributors (e.g. `AddVirtualJson` added in a downstream module) win over earlier ones from upstream modules. That is how downstream apps override framework strings.

## Dynamic / external resources

`IExternalLocalizationStore` lets language management write resources to the database and expose them through the same `AbpStringLocalizerFactory`. The factory always checks the static catalog first; if nothing is found, it calls `ExternalLocalizationStore.GetResourceOrNullAsync(name)` and caches the result in `LocalizerCache`. Contributors returned by an external store should set `IsDynamic = true` so the per-call `includeDynamicContributors` flag can exclude them when needed.

## Virtual-file integration

Localization files live in the same virtual file system as views, scripts, and other static assets. `AbpLocalizationModule` embeds the framework's own files (`Volo.Abp` / `Volo/Abp`). Downstream modules embed their own assembly and call `AddVirtualJson("/Localization/Foo")` so files under `Localization/Foo/*.json` in the assembly get picked up. See `/core/virtual-file-system`.
